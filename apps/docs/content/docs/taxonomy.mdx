FusionKit documentation is organized by reader intent. Each page should answer one kind of question: what to read first, how to complete a task, how the system works, what a reference surface contains, how to operate the project, or how to call an API.

Use this page when you are deciding where to go next or where a new documentation entry should belong.

## The taxonomy

| Category | Reader question | Page behavior |
| --- | --- | --- |
| Orientation | Where am I, and what should I read next? | Explain scope and route readers to the right section. |
| Task guides | How do I complete this workflow? | Provide commands, expected output, and the next troubleshooting step. |
| Concepts | How does the system fit together? | Explain mental models, boundaries, and architecture. |
| Reference | What exactly does this surface contain? | Document commands, fields, packages, routes, examples, and stable interfaces. |
| Operations | How do I run, publish, or maintain this system? | Name workflows, scripts, credentials, verification, and recovery paths. |
| APIs | How do clients call FusionKit? | Describe runtime routes and generated contracts. |

## Entry justification

Every public documentation entry should have a reason to exist. A page belongs in the site when it helps a user install FusionKit, run a session, configure a repo, understand product behavior, integrate an SDK, operate a supported surface, run an example, or call an API.

If a page is mostly about implementation ownership, generated code, release internals, benchmark maintenance, or historical design, it belongs in the repository maintainer docs rather than the public site. If a page is a temporary plan, it belongs outside visible docs.

## How to choose a page

Start with [Installation](/docs/getting-started/installation) if you have not run FusionKit before. Use [Quickstart](/docs/getting-started/quickstart) when you want the shortest path to a fused coding session. Use [Inference endpoint](/docs/getting-started/inference-endpoint) when you want curl or SDK access without a coding harness. Use [Models and panels](/docs/cli/models-and-panels) when choosing providers, local MLX, mixed panels, or budgets.

Use [Product scope](/docs/concepts/product-scope) when package names or retained platform packages are confusing. Use [Architecture](/docs/concepts/architecture) when debugging a full session. Use [API Reference](/docs/api) when writing a client. Use [Troubleshooting](/docs/cli/troubleshooting) when a command fails.
