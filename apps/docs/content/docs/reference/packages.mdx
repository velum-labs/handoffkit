FusionKit is a pnpm + TypeScript workspace plus a uv + Python workspace. The packages below are the product surface used by the shipped `fusionkit` journey. Legacy governance packages live under `legacy/` and are outside the public user path.

## Product TypeScript packages

| Package | Responsibility |
| --- | --- |
| `@routekit/cli` | Independent `routekit` executable for router config, endpoint/account management, gateway serving, and coding-tool launch. |
| `@routekit/config` | Reusable RouterConfig discovery, layered loading, validation, and atomic writes. |
| `@routekit/router` | Reusable embedded router composition and lifecycle. |
| `@fusionkit/config` | Fusion-only v4 schema, endpoint-ID ensembles, prompt loading, and atomic writes. |
| `@fusionkit/cli` | Fusion-only command line. It composes RouteKit SDKs but never depends on `@routekit/cli`. |
| `@routekit/gateway` | Product-neutral HTTP gateway, wire dialects, provider egress, endpoint catalogs, pooling, and single-call provenance. |
| `@routekit/accounts` | Subscription credentials, account pools, provider relays, proxy clients, and managed CLIProxyAPI lifecycle. |
| `@fusionkit/gateway` | Fusion front-door orchestration, sessions, aggregate cost accounting, trajectories, and managed local-model lifecycle. |
| `@fusionkit/ensemble` | Panel orchestration, worktrees, runtime-kernel workflows, judge adapters, and advanced harness tooling. |
| `@routekit/tools` | Product-neutral launcher, driver, and capability registry. |
| `@routekit/tool-registry` | Canonical product-neutral registry consumed by both the RouteKit and FusionKit CLIs. |
| `@routekit/tool-codex`, `@routekit/tool-claude`, `@routekit/tool-cursor`, `@routekit/tool-opencode` | Canonical launcher and driver for each coding tool. |
| `@fusionkit/protocol` | Model-fusion contracts, schemas, OpenAPI bindings, trace records, and generated SDK bindings. |
| `@fusionkit/workspace` | Git capture, worktree materialization, output collection, and divergence-safe pull helpers. |
| `@fusionkit/adapter-ai-sdk` | Managed MLX server and AI SDK model adapters used by local-model paths. |
| `@fusionkit/kernel` | Runtime-kernel graph, scheduling, budget, streaming, and outcome primitives. |

## Python packages

| Package | Responsibility |
| --- | --- |
| `fusionkit` | Internal PyPI runtime that installs only `fusionkit-sidecar`; the Node package owns the user-facing `fusionkit` executable. |
| `fusionkit-core` | Provider-neutral fusion engine, RouteKit client, sidecar config, judge, run manager, artifacts, and metrics. |
| `fusionkit-server` | Internal FastAPI sidecar with health, trajectory-fusion, and native-run routes; it has no public chat/model front door. |
| `fusionkit-mlx` | Optional MLX launcher utilities. |
| `fusionkit-evals` | Separately installed `fusionkit-bench`, public reports, prompt tuning, Pareto analysis, and hill climbing. |
| `fusionkit-testkit` | Unpublished RouteKit-upstream simulator and sidecar process fixtures. |

## Legacy archive

The Warrant plane, runner, SDK, handoff SDK, compute adapter, and session backends are preserved under `legacy/`. They are not part of the normal `fusionkit codex` / `fusionkit serve` product path.

## Adding a new tool

Each coding tool is its own workspace package implementing a `ToolIntegration`:

1. Create `packages/tool-<name>/` from an adjacent tool package.
2. Export a `ToolIntegration` with a `launch(ctx)` implementation.
3. Add it to `toolIntegrations` in `@routekit/tool-registry`.

That one canonical entry wires the tool into both CLIs and internal harness
composition through a neutral `ToolLaunchSpec`; consumers do not keep parallel
imports or lists. The existing integrations are documented under [Tools](/docs/tools).
