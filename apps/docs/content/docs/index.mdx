## What is fusionkit?

`fusionkit` puts **real model fusion** behind your coding agent. On every prompt it:

1. Spins up a **panel** of models (a cloud trio by default, or a local MLX trio).
2. Has each panel model produce a **real candidate** by running a coding harness
   over your repo.
3. Runs a **judge** that synthesizes the answer your coding agent (Codex, Claude
   Code, or Cursor) actually executes.

You run one command and your agent is transparently backed by the fused panel:

```bash
npm install -g @fusionkit/cli
cd your-project        # a git repo
fusionkit doctor       # verify prerequisites
fusionkit codex        # launch Codex backed by the panel
```

## Two ways to use it

fusionkit ships as a single CLI on top of a governed execution platform. Most
people start with the CLI; teams building their own agent products reach for the
platform packages.

## The platform underneath

Both surfaces sit on the same governed execution and provenance plane (the
project's internal name is **Warrant**). Its two core objects are the **run
contract** (a signed authorization to execute under stated conditions) and the
**receipt** (a signed, offline-verifiable record of what actually happened).
Continuation — handing local work to a governed runner and pulling it back — is
built from those same primitives and shipped as the [handoff
SDK](/docs/sdks/handoff-sdk).

<Callout type="info">
  Two packages share the name "fusionkit": the npm CLI (`@fusionkit/cli`) and the
  Python synthesizer (`fusionkit` on PyPI), which is fetched automatically via
  `uvx`. `fusionkit --version` prints both.
</Callout>
