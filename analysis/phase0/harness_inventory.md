# Phase 0 harness capability inventory

Scope: verify what `fusionkit-evals` can execute today for C3/M5 calibration, against the eight primary taxonomy domains in `docs/fusion/capability-index-spec.md` section 8.

## Executive verdict

- **C3 cannot run SWE-bench-style repo_bugfix tasks with official per-instance grading using only the in-repo harness today.** `public_bench.py` knows about `swe-bench-pro`, but it delegates execution to an operator-supplied external runner adapter; no official runner is implemented in this repo. Evidence: `PUBLIC_BENCHMARK_SUITES` lists `swe-bench-pro`, `PUBLIC_BENCHMARK_INFO` says the verifier is the official secure harness, and `CommandExternalBenchmarkExecutor` only shells out to an adapter command. (`python/fusionkit-evals/src/fusionkit_evals/public_bench.py:38`, `python/fusionkit-evals/src/fusionkit_evals/public_bench.py:87`, `python/fusionkit-evals/src/fusionkit_evals/public_bench.py:303`)
- **The repo task path that does exist is HandoffKit contract ingestion, not official SWE-bench grading.** `FusionBenchRunner.run_tasks` only dispatches `model_fusion` and `harness_coding`; `harness_coding` skips unless a `HandoffKitExecutor` is configured, then joins `harness-run-result`, candidate, model-call, tool, artifact, and receipt records. (`python/fusionkit-evals/src/fusionkit_evals/fusion_bench.py:294`, `python/fusionkit-evals/src/fusionkit_evals/fusion_bench.py:318`, `python/fusionkit-evals/src/fusionkit_evals/fusion_bench.py:488`)
- **`CandidateBank` is not arbitrary-task storage.** It is `task_id`, `prompt`, `tests`, optional `difficulty`, plus generated candidates with a boolean pass flag; `build_candidate_bank` generates panel candidates, extracts code, and verifies code against tests. (`python/fusionkit-evals/src/fusionkit_evals/candidate_bank.py:34`, `python/fusionkit-evals/src/fusionkit_evals/candidate_bank.py:99`, `python/fusionkit-evals/src/fusionkit_evals/candidate_bank.py:140`)
- **The default sandbox is local subprocess execution, with optional Docker if installed/configured.** `LocalSandbox` runs Python `sol.py` in a temp dir with scrubbed env and POSIX resource limits; `DockerSandbox` constructs a `docker run --network none --read-only` command and `build_sandbox` selects `local` by default or `BENCH_SANDBOX`. (`python/fusionkit-evals/src/fusionkit_evals/sandbox.py:73`, `python/fusionkit-evals/src/fusionkit_evals/sandbox.py:152`, `python/fusionkit-evals/src/fusionkit_evals/sandbox.py:228`)

## Implications for C3/M5

- **C3 can run algorithmic calibration today** with the existing candidate-bank/LiveCodeBench-style path, assuming provider credentials and a selected/frozen task list.
- **C3 can run repo_bugfix only as a bespoke HandoffKit/HandoffKit-compatible patch-and-test pilot, not as official SWE-bench-style per-instance grading.** That may still be useful for transfer risk, but it should be labeled tier-CAL internal harness data, not SWE-bench evidence.
- **C3 cannot cover all eight taxonomy domains with deterministic grading today.** To cover every primary domain, build or wire external adapters for frontend UI, SQL/data, backend DB/API, terminal containers, security, and migration/refactor suites.
- **M5's assumption that `fusionkit-evals` can freeze a `CandidateBank` across domains is too broad.** The bank works for prompt+tests code tasks and the polyglot adaptation; HandoffKit/public benchmark paths produce run rows/envelopes, not `CandidateBank` tasks with `prompt/tests/candidates`.
