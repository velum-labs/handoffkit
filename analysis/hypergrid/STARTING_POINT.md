## 2. Benchmark pool — a plan deviation, flagged

The plan assumed cutoff candidates {2025-06-01, 2025-10-01, 2026-01-01}.
**Those are impossible:** `livecodebench/code_generation_lite` `release_v6`
(latest, last modified 2025-06-05) ends at contest date **2025-04-06**.
Eligible pool (stdin-only public tests, no starter code, medium/hard, dedup by
question_id):

| cutoff | eligible | medium / hard |
|---|---|---|
| 2024-08-01 | 220 | 68 / 152 |
| 2024-10-01 | 164 | 50 / 114 |
| 2025-01-01 | 86 (the prior exec-select run's slice) | 26 / 60 |
| 2025-02-01 | 64 | 19 / 45 |

**Draft split: window ≥ 2024-08-01 (220 tasks), seed 20260712 → dev 110,
holdout 70, spare 40** (`manifests/dev.txt`, `manifests/holdout.txt`,
`manifests/spare.txt`, provenance in `manifests/manifest_meta.json`). The
2025-01-01 cutoff the repo used before cannot support disjoint dev+holdout at
the planned sizes (86 < 180).

**Known limitation to decide at start:** every task in the window predates the
training cutoffs of the 2025-h2/2026-h1 registry models, so absolute rates
will be contamination-inflated for *all* cells. The comparison is still fair
(anchors are newer-generation too and equally exposed), and the gap-to-SOTA
objective is a paired comparison on identical tasks — but rung-3 claims
("matches SOTA") must carry this caveat. Mitigations if the supervisor deems
the slice too soft at gen 0: restrict to the hard-only subset (152 tasks ≥
2024-08-01), or swap in a newer LCB release if one ships.

## 4. Generation-0 grid (materialized, not run)

From `gen0.py` via the real hyperkit `Experiment` contract — 13 cells, 1,430
shards at the 110-instance screen rung; kernel-probe templates (judge-synth,
judge-select, self-MoA, exec-select) are defined as data and appended by
`hyperkit extend` after the screen names the top-2 complementary solos.

Cost preview (`grid_preview.md`): solo screen + anchors ≈ **$58** (of which
$38.50 is the two anchors), kernel probes ≈ **$10**, generation-0 total ≈
**$68 vs the $65 phase gate** — marginally over. Options at start: run anchors
at the 60-instance rung first (saves ~$17, promotes to 110 later via the
shard-reuse protocol), or accept a $70 gate. Recommendation: anchor at the
60-rung first; the gap computation only needs anchor results on instances the
compound also ran.

## 5. Prior-work baselines this climb starts from

  59.3% vs 47.7% best solo, McNemar 10–0 — the kernel to beat, now as a

  supervisor prune rules in the plan.

## 6. Gaps that must be built before the first billed shard (plan Phase 0b/1)

1. `hyperkit/adapters/livecodebench.py` — Docker-free adapter (exists: nothing;
   the two shipped adapters both need Docker).
2. `run_instance(..., params)` seam in the `BenchmarkAdapter` protocol +
   orchestrator (exec-select kernel params are cell coordinates).
3. Parallel `LocalComputeBackend` (`max_workers`); it is strictly sequential
   today — 1,430 network-bound shards would be serial.
4. Local controller loop (filesystem → CellSnapshots → OTLP) + OTLP-enabled
   Prometheus config for the Grafana stack; Docker install with Firecracker
   workarounds.
5. `.cursor/skills/hypergrid-hillclimb/` skill + `supervisor.py` helper.
6. Preflight run + `preflight.md` (billed smokes are part of it).
