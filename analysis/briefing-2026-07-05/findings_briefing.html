One sentence: We are building a tool that runs several cheap open-source AI coding models at once and merges their answers into one — and we spent ~$34 running controlled experiments to learn where that idea can work, where it cannot, and what to do next before we launch.

Plain English — the whole story in five bullets

  • What we're building: FusionKit (part of the HandoffKit repo) — a developer CLI. You send one coding question; behind the scenes 2–4 models answer in parallel; a "judge" merges them; you get one reply. Think "committee of models" instead of one expensive model.
  • Why it might work: Different models fail on different tasks. If Model A passes tasks 1–3 and Model B passes 3–5, a smart merger could pass more tasks than either alone. We measured this "extra room" (headroom) on public benchmarks — it is real (+8–12pp).
  • What we learned that surprised us: You cannot pick which models to put on the committee by reading public leaderboard scores. That strategy lost in every rigorous test (0 wins out of 14). You have to run your own small experiments (~$6–17 each) to choose panel members.
  • Where we would launch: On open-source model fields where no single model dominates ("peer-shaped"). Repo bugfix (fixing real GitHub bugs) looks best: big headroom (+17pp), real user demand, but we need engineering work before we can grade it ourselves.
  • The open bet: Our merge step ("synthesis") once scored above the theoretical best-of-committee ceiling — suggesting merging can create new correct answers, not just pick one. That happened once, with caveats. A ~$15 pilot will test whether it replicates.

Product context — what we are trying to build

The product (FusionKit): a command-line tool (CLI) where a developer types a coding task — "fix this bug," "write this function" — and the tool sends that task to several AI models at once. Those models are mostly open-source (OSS): publicly available weights/APIs from vendors like DeepSeek, Qwen, and Moonshot, typically much cheaper per token than closed frontier models (OpenAI GPT, Anthropic Claude). Each model returns a candidate answer. A separate judge model reads all candidates and produces one merged answer. The developer sees a single response, as if one model answered.

Why "ensemble" instead of one big model? Frontier models (gpt-5.5, claude-opus) score highest on benchmarks but cost more per solved task. Our bet: a committee of cheaper OSS models, merged well, might get close enough on score while costing much less — a win on both axes (quality and price), which economists call a Pareto improvement.

The business claim we want to make: not "we beat every leaderboard" — frontier labs will always out-saturate benchmarks — but: "For repo bugfix [or another domain], our fused OSS panel solves X% of tasks at $Y per solve, vs frontier model Z at $W per solve." Score and dollars per solved task, side by side. Our panels are OSS-first (decision D8): closed models like gpt-5.5 are price anchors and baselines we compare against, not members of the committee we sell.

What is a "benchmark" in this doc? A fixed set of coding tasks with an automatic grader (tests that pass/fail). Example: SWE-bench Verified = real GitHub issues where success means your patch makes the project's test suite pass. We measure pass rate = tasks passed ÷ tasks attempted. On 60 tasks, 27 passes = 45.0%.

The open engineering question: (1) Which OSS models to combine? (2) For which kind of coding work (domain)? (3) Does our merge pipeline actually convert theoretical headroom into real pass-rate gains? This briefing reports progress on those three questions after ~$34 of measured experiments.

What this briefing is not: a final launch claim, a benchmark submission, or a promise that fusion always wins. Several findings are deliberately negative ("don't fuse here", "public data can't rank panels") — those protect us from launching the wrong thing. Negative results are product features, not failures.

Glossary — terms used throughout

Every acronym and metric is defined here. In the findings sections, look for blue How to interpret boxes and gray worked examples.

Ensemble / panel
A set of K models (usually 2–4) that all attempt the same task. "Panel" = the members who answer. "Ensemble" = members plus the judge/merger. Example panel: DeepSeek-R1 + Qwen3 + DeepSeek-V3.
Fusion / synthesis
How K candidate answers become one answer. Selection = a judge picks the best candidate verbatim ("I'll take Model B's answer"). Synthesis = a judge reads all candidates and writes a new answer that can combine parts of several ("take A's imports and B's loop logic"). Our product does synthesis — that is the potential moat.
Judge
A model (often separate from panel members) that reads all panel outputs and produces the final merged answer. Must not be a panel member when measuring capture fairly — otherwise it could favor its own style.
Pass rate / pass@1
Fraction of tasks where the model (or fused output) passes the benchmark's grader. On 60 tasks, 27 passes = 45.0%. Higher is better. This is our main quality metric.
pp (percentage points)
Absolute difference between two pass rates. 45% vs 30% = +15pp — not "+50% relative improvement." We always use pp so comparisons are unambiguous.
Task / item / row
One problem in a benchmark (one bug to fix, one function to write). We often run n=60 tasks per experiment for statistical power.
Harness
Our software that sends tasks to models, collects answers, runs graders, and logs results. "Our harness" = we control prompts, token limits, and grading — not a third-party leaderboard.
Grader
Automatic test that decides pass/fail (unit tests, diff against golden patch, etc.). No human reads each answer in these experiments.
Oracle / panel ceiling
Hypothetical perfect referee: on every task, if any panel member passed, count it as passed. Example: Model A passes tasks 1–20, Model B passes 15–35 → oracle passes 35/60 = 58.3% even if A alone = 33% and B alone = 33%. Measures potential — no real fusion system achieves the oracle unless capture ≥ 100%.
Headroom
Oracle minus best single member's pass rate. Example: best member 45%, oracle 62% → headroom = +17pp. That is the maximum uplift fusion could add if errors were perfectly complementary and merging worked perfectly.
Capture rate
How much headroom the actual fused pipeline realizes: (fused − best member) ÷ (oracle − best member). Example: best=45%, oracle=62%, fused=53% → capture = (53−45)/(62−45) = 8/17 ≈ 47%. 100% = matched oracle. >100% = beat oracle (only possible with synthesis, not selection).
Complementary errors
When models fail on different tasks — A fails where B passes and vice versa. Required for headroom. If all models fail the same tasks, fusion cannot help.
Peer field vs lopsided
Peer = several models score close together (e.g. 30% vs 28%) → ensembles can help. Lopsided = one model dominates by 15–20+pp (e.g. 80% vs 45%) → honest answer is "use the single winner; don't pay for a committee."
Layer-1 / public prior
Evidence mined from public benchmark dumps others published (SWE-bench submissions, LLMRouterBench, Terminal-Bench). Costs $0. Good for shortlisting ("these models are plausible") and vetoing redundant pairs; not authoritative for final panel picks (see §3).
Calibration / Layer-CAL
Runs we execute ourselves: we pick tasks, call APIs, pay the bill, run our harness. This is where launch-grade, defensible numbers come from. Typical cost ~$6–17 per small round, ~$30–50 per flagship pilot.
Checkpoint (C0, C1, C2…)
Named experiment in Phase 0. C1 = "is headroom real?" C2 = "can public data pick panels?" C3 = transfer to our harness. C2V = replicate C2 with different objective. Numbers like "C3-R16K" = C3 rerun with 16k token budget.
Pre-registration / prereg
We wrote the test plan (which models, how many tasks, pass/fail rules, spend cap) before looking at results and committed it to the repo. Prevents cherry-picking or moving goalposts after seeing data.
Held-out / train-test split
Split tasks in half: use first half to choose a panel, second half to score it. The scorer never saw the second half during selection — mimics "will this work on new tasks?"
Top-K baseline
The dumb competitor: pick the K models with highest average pass rate on training tasks. C2 asks: can smarter selection beat this simple rule? Answer: no, reliably.
Truncation
Model hit the maximum output token limit before finishing its answer (answer cut off mid-code). Truncated answers usually fail grading for mechanical reasons, not because the model "didn't know." We refuse pass-rate claims when >10% of rows truncate.
Token budget
Maximum tokens the model may generate in one response. "32k budget" = up to ~32,000 tokens of output. Thinking models use tokens on internal reasoning before the visible answer — they need larger budgets.
Wilson CI / 95% CI
Confidence interval on a pass rate — the range we'd expect if we repeated the experiment. Written as [low, high], e.g. [33.1, 57.5]. Narrow = more certainty. For differences, if the interval crosses zero, the result is inconclusive.
Frontier / price anchor
A closed, expensive model (gpt-5.5, claude-opus) used to compare score and $/solve on the launch card — not a panel member in our OSS-first strategy.
OSS
Open-source or open-weights models (DeepSeek, Qwen, Kimi via API, etc.) — typically cheaper; our product panels use these.
$/solve
Total API cost ÷ number of tasks passed. If you spend $30 and pass 15/60 tasks, $/solve = $2.00. Used for Pareto claims alongside pass rate.
Evidence card
Public-facing summary table: domain, panel, pass rates, $/solve, validity notes (truncation audit), frozen config hash — what we'd put on a marketing/docs page.
Decision D1–D11
Binding conclusions recorded in capability-index-program.md after each experiment (e.g. D2 = public data can't rank panels; D8 = OSS-first; D10 = repo bugfix flagship; D11 = kimi-thinking excluded).
SWE-bench / SWE-V
Software engineering benchmark: fix real GitHub repo bugs; grader runs project tests. "Verified" = curated subset. Repo bugfix domain in our scan.
LiveCodeBench (LCB)
Competitive programming / algorithmic coding benchmark — "write a function that passes hidden tests." Our algorithmic slice.
Terminal-Bench
Agentic tasks in a terminal environment (shell commands, file edits). Terminal/agentic domain.
Lineage veto
Rule in OSS scan: don't put two models on a panel if one is a fine-tune of the other (they make correlated errors → less headroom).
System vs model-level
Model-level = single model API call. System-level = full agent scaffold (tools, retries, multi-step) as submitted to SWE-bench. Both appear in repo bugfix scan rows.
Worked example — headroom and capture: Panel of 3 models on 60 tasks. Best member alone: 27/60 = 45%. Oracle (any member passes): 38/60 = 63.3%. Headroom = 18.3pp. If fusion passes 32/60 = 53.3%, capture = (53.3−45)/(63.3−45) ≈ 45% — we captured less than half the theoretical room.

How to read the charts & confidence tags

Every chart uses the same bar layout: label (left) → colored bar (middle, width = magnitude) → number or CI (right). Bar width is scaled to a max shown in each section's footnote — bars are for comparison within a chart, not absolute pixel = absolute percent.

Blue bars — opportunity numbers: oracle, headroom, or spend. These are potential or cost, not "our product already achieved this score."
Green — valid measurement (truncation OK) or peer-shaped field where fusion is worth trying.
Gray — closed frontier models or baselines we compare against; not panel members we sell.
Red / hatched — invalid measurement (truncation), clear loss, or explicit "don't fuse" signal.
Purple — actual fused/synthesized output — the product deliverable we are trying to optimize.
Gray CI bands (§3) — 95% confidence interval on a difference. Vertical line = zero (no difference). Entire band right of zero = statistically solid win.

Stacked bars in §5: The bar shows the panel oracle pass rate (full width). The dashed vertical line marks where the best single member alone would sit — everything to the right of the line is headroom. Example: bar to 45.6%, dashed line at 28.6% → headroom +17.0pp.

Numbers in brackets [a, b]: 95% confidence interval. We are 95% confident the true value lies in that range. Wider interval = fewer tasks or noisier measurement.

Confidence tags in §11:

High = pre-registered or independently recomputed from raw files; we'd stake a product decision on it.   Medium-high = strong but from public data we didn't run ourselves, or only one domain tested so far.   Medium = real signal, but an open caveat remains (e.g. contamination check not done yet).   Directional = interesting one-off result; not yet replicated under the clean protocol we'd use for launch.

2Complementary errors are real — the raw material exists

Checkpoint C1 (public per-task data, three independent sources): the best 2–3 system panels beat their own best member's ceiling by a wide, statistically solid margin. Oracle headroom = extra tasks solvable if a perfect referee always picked a passing member.

How to interpret

What we measured: On three public benchmarks, we took the best 2–3 systems (full agent setups, not just raw models) and computed oracle headroom — how many extra tasks a perfect referee could solve vs the best single member.

What the bars mean: +9.0pp on SWE-bench Verified means: on that benchmark, the best panel's oracle score is 9 percentage points higher than its best member alone. The CI [+6.5, +12.5] means we're confident the true headroom is in that range — it is not noise.

What this does NOT mean: Our FusionKit product does not yet achieve +9pp on SWE-bench. We have not run fused output on these benchmarks at launch scale. This only proves the raw material exists — different strong systems fail on different tasks.

So what: If headroom were ~0 everywhere, fusion would be pointless (every model would fail the same tasks). It is not ~0. The remaining question is capture (§7): can our merge pipeline actually realize this headroom?

3…but public data cannot pick the panel (settled, twice)

Checkpoints C2 + C2V: choose a panel on half the tasks, score it on the held-out half, against the dumb baseline "take the top-K models by average score." Complementarity-smart selection never reliably won — and lost outright on SWE-bench Verified. Tested with two selection objectives across 7 datasets.

How to interpret

The experiment (C2 + C2V): Split each benchmark's tasks in half. On the first half ("train"), pick a panel using public per-task data. On the second half ("held-out"), score that panel. Compare against the baseline: "just pick the K models with the highest average score on train."

Reading the CI plots: Each row shows the difference (smart selection minus dumb baseline) on held-out tasks. The gray/red band is the 95% CI; the black tick is the best estimate; the vertical line is zero. If the entire band is to the right of zero, smart selection wins. If entirely to the left, it loses. If the band crosses zero, we call it inconclusive — not enough evidence either way.

SWE-V · K=2 loses (−3.0pp): On SWE-bench Verified with 2-model panels, complementarity-smart selection was 3 points worse than "pick the top-2 by average score" on tasks it had never seen. That is the opposite of what we hoped.

0 / 14 wins: Across 14 dataset × panel-size × objective combinations, smart selection never achieved a statistically solid win. Hence decision D2: public data may shortlist and veto, but final panel picks require our own calibration runs (~$6–17 each — cheap compared to a wrong launch).

Analogy: Public data tells you which job candidates interviewed well on past problems — useful for building a shortlist. It does not tell you which team of 3 will outperform on new problems. We tried; the "smart team picker" lost to "hire the 3 highest GPA" on held-out work.

4Field shape decides whether to fuse — and OSS fields are peer-shaped

Fusion only pays when several models are comparably strong ("peer field"). Against closed frontier models, our algorithmic slice is hopelessly lopsided. Exclude closed models — the OSS-first product decision (D8) — and every covered domain becomes peer-shaped.

How to interpret

Core idea: Fusion only makes sense when multiple models are similarly strong. If one model beats everyone else by a huge margin, the honest product recommendation is "use that one model" — running a committee adds cost without score benefit.

Left panel — lopsided (don't fuse for score): Our own 60-task algorithmic benchmark. gpt-5.5 (closed frontier) passes 48/60 = 80%. Best OSS model passes ~18/60 = 30%. Gap = +35pp → lopsided. Even a perfect OSS fusion cannot beat gpt-5.5 on this slice. Our OSS product claim here is cost ("similar-ish quality, much cheaper"), not "higher pass rate."

Right panel — peer-shaped (fusion plausible): When we look only at OSS models on five domain types (from public data), the gap between #1 and #2 is small (+1.7pp to +9.2pp). None exceed our 15–20pp "lopsided" threshold. Multiple OSS models win different tasks → headroom exists.

Why two panels side by side: Same data can look lopsided or peer-shaped depending on who is in the comparison set. Decision D8 (OSS-first) means we compare OSS panels against OSS peers for fusion design, and against frontier models only for price anchoring.

6Measurement integrity: the truncation ladder

Our single most repeated lesson: thinking-style models produce garbage numbers unless completion budgets are huge — and published benchmark scores rarely disclose this. We enforce: >10% truncated rows ⇒ pass rate refused.

How to interpret

Why this section exists: Some models — especially "thinking" models — use most of their output token budget on internal reasoning before writing the visible answer. If the budget runs out, the answer is cut off mid-sentence/mid-code → fails grading → looks like the model is bad when it was actually incomplete.

Our rule (D11): If more than 10% of tasks truncate (10+ out of 60), we refuse to publish a pass rate for that model at that budget. The score is unmeasurable, not "low."

Left — kimi-k2-thinking: Pass rate appears to rise as budget increases (11.7% → 28.3%), but truncation stays high (52/60, 42/60, 31/60). You are measuring "how much answer fit in the box," not ability. Excluded from single-shot panels.

Right — claude-sonnet-4-6: At 16k budget, 14/60 truncate → invalid. At 32k budget, 0/60 truncate → valid 45.0%. First trustworthy number for this model on our slice.

Product angle: Most leaderboard entries do not report truncation rates. We can — that is a credibility differentiator on evidence cards ("valid @ 32k, 0% truncated").

7The wildcard: synthesis once beat the theoretical selection ceiling

In the original C3 replay, the fused answer stream scored above the frozen bank's oracle — impossible for any pick-the-best-candidate scheme, possible only if synthesis combines partial answers into new passing ones.

How to interpret

Setup: On 60 algorithmic tasks, we already had frozen panel member outputs from Phase 0 (no new member API cost). We re-ran only the judge/merger on those same outputs and graded the fused answers — a "replay."

Three bars explained:

① Best single member (28.1%) — the highest pass rate any one panel model got alone.
② Oracle ceiling (35.1%) — if a perfect judge always picked a passing member's answer verbatim, 35.1% would pass. No "pick the best" scheme can exceed this.
③ Fused output (38.6%) — what our synthesis judge actually produced. It beat the oracle → 150% capture.

Capture math: (38.6 − 28.1) ÷ (35.1 − 28.1) = 10.5 ÷ 7.0 = 150%. We gained 10.5pp over the best member; the oracle gap was only 7.0pp — so we captured more than 100% of theoretical headroom. That implies synthesis created new passing answers by combining partial solutions, not just picking one.

Why "directional" not "proven": Fallback judge used; same tasks as Phase 0 (not fresh hold-out); not yet replicated under capture-pilot-1 prereg. If it replicates cleanly, the product story upgrades from "cheaper committee" to "committee that can beat any single member."
